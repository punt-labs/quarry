\section*{Press Release}

\prsection{Summary}

Today, the Quarry project announced the general availability of Quarry, a free,
open-source tool that gives Claude instant access to everything on your computer.
Quarry indexes PDFs, scanned documents, spreadsheets, presentations, web pages,
notes, source code, and images into a local semantic search engine, then exposes
that knowledge to Claude Code and Claude Desktop through the Model
Context Protocol (see \faqref{faq:vs-projects} and \faqref{faq:vs-folders}).
Unlike uploading files to Projects or pointing your LLM at folders, Quarry lets
Claude search your entire knowledge base by meaning---across every format, with
no manual organization required.

\prsection{Problem}

People who use Claude for serious work---refactoring codebases, analyzing
research, drafting from reference materials---hit the same wall every session.
They spend 5--15 minutes gathering context before they can ask their real
question. They upload PDFs to Projects. They copy-paste from old documents. They
point Claude Code at directories and hope the right files get picked up. When the
context window fills, they start over.

A developer with ten years of reference PDFs and design documents cannot make any
of that knowledge available to Claude without manually curating it first. A
researcher with hundreds of papers must decide which three are relevant before
the conversation starts. The LLM can only reason about what it can see---and
most of what you know is invisible to it.

\prsection{Solution}

Quarry works in three steps: install, ingest, search. Run
\texttt{uv tool install punt-quarry} and \texttt{quarry install}. Point it at
your files---\texttt{quarry ingest report.pdf}, or register an entire directory
with \texttt{quarry register \texttildelow/Documents/research}. From that moment,
Claude can search your indexed knowledge by meaning rather than keywords.

Every format is handled automatically. Text PDFs are extracted. Scanned documents
and images are OCR'd---locally, with no cloud account. Spreadsheets are serialized
to LaTeX tabular format. Presentations are split slide-by-slide. HTML pages are
stripped of boilerplate and converted to Markdown. Source code is parsed into
semantic units using Tree-sitter. Markdown, LaTeX, and DOCX files are split by
section structure. The result: chunks Claude can retrieve, with metadata
indicating the document, page, content type, and source format.

Because Quarry runs entirely on your machine, your documents never leave your
laptop. No cloud service, no API key, no subscription.

As a Claude Code plugin, Quarry goes further---it captures knowledge
automatically. Web pages Claude fetches during a session are ingested into your
knowledge base. Conversation transcripts are preserved before context compaction.
Your project directory is registered the moment a session starts. Every hook is
fail-open and non-blocking: you work normally, and the knowledge base grows.

\prsection{Getting Started}

Getting started takes under five minutes:

\begin{enumerate}[leftmargin=2em]
  \item \textbf{Install:} {\small\texttt{uv tool install punt-quarry \&\& quarry install}}
        downloads the embedding model and configures Claude Code and Claude
        Desktop automatically.
  \item \textbf{Ingest:} \texttt{quarry register \texttildelow/Documents/work
        \&\& quarry sync} indexes an entire directory. Or ingest individual files
        with \texttt{quarry ingest report.pdf}.
  \item \textbf{Search:} Open Claude Code or Claude Desktop and ask a question.
        Claude calls Quarry's search tool automatically.
\end{enumerate}

\section*{Frequently Asked Questions}

% ── External FAQs (Customer-facing) ──────────────────────────────────────────

\subsection*{External FAQs}

\begin{faqpair}{What is Quarry and who is it for?}\label{faq:what-is-quarry}
  Quarry is a local semantic search engine for your files that integrates with
  Claude through MCP. It is for people who use Claude Code or Claude Desktop
  for real work---writing, coding, research, analysis---and are tired of
  manually uploading files before every session. If you have documents on your
  hard drive that you wish Claude could see, Quarry is for you.
\end{faqpair}

\begin{faqpair}{How is this different from uploading files to Claude Projects?}\label{faq:vs-projects}
  Projects require you to select and upload files before every conversation, the
  context window limits how many you can include, and when you start a new
  conversation, you start from scratch. Quarry inverts this: index your files
  once, and Claude searches them on demand during any conversation. You don't
  decide what's relevant---Claude does. There is no context window limit because
  Quarry returns only the relevant passages, not entire files.
\end{faqpair}

\begin{faqpair}{How is this different from pointing Claude Code at a folder?}\label{faq:vs-folders}
  When Claude Code reads a directory, it reads files sequentially, consuming
  context window with every file it opens. For a folder with 50 documents, Claude
  Code may exhaust its context before finding the relevant passage. Quarry
  pre-indexes the content and returns only matching chunks, leaving the context
  window for reasoning rather than reading.
\end{faqpair}

\begin{faqpair}{What file formats does Quarry support?}\label{faq:formats}
  PDF (both text and scanned/image pages), images (PNG, JPG, TIFF, BMP, WebP),
  spreadsheets (XLSX, CSV), presentations (PPTX), HTML, text files (Markdown,
  LaTeX, plain text, DOCX), and source code in 30+ languages. Every format is
  converted to text optimized for LLM consumption. Spreadsheets and presentation
  tables are serialized to LaTeX to preserve tabular structure. Scanned documents
  and images are OCR'd locally---no cloud account needed.
\end{faqpair}

\begin{faqpair}{Does my data leave my computer?}\label{faq:privacy}
  No. Quarry runs entirely on your machine. The vector database (LanceDB), the
  embedding model (snowflake-arctic-embed-m-v1.5 via ONNX Runtime), and the OCR
  engine (RapidOCR) all run locally. No data is sent to any cloud service. The
  only optional cloud component is AWS Textract for higher-quality OCR of
  degraded scans, and that is opt-in.
\end{faqpair}

\begin{faqpair}{How much does it cost?}\label{faq:cost}
  Free. Quarry is open source under the MIT license. There is no paid tier, no
  usage limits, and no telemetry. The embedding model download is approximately
  500\,MB; after that, everything runs offline.
\end{faqpair}

\begin{faqpair}{Can I use Quarry with tools other than Claude?}\label{faq:other-tools}
  Quarry exposes a standard MCP server over stdio. Any MCP-compatible client can
  use it. Today that includes Claude Code and Claude Desktop.
  As other LLM tools adopt MCP, Quarry will work with them without modification.
  The CLI also works standalone for direct search without any LLM.
\end{faqpair}

\begin{faqpair}{How does search quality compare to keyword search (Spotlight, grep)?}\label{faq:search-quality}
  Quarry finds passages by meaning, not by matching words. Searching for
  ``authentication logic'' finds a function called \texttt{verify\_credentials}
  even though the words don't overlap. This is the same retrieval approach used
  by commercial RAG systems, running locally with a 768-dimension embedding
  model.
\end{faqpair}

\begin{faqpair}{Do I have to manually index everything?}\label{faq:passive-capture}
  No. The CLI and MCP server require explicit commands, but the Claude Code
  plugin captures knowledge passively through hooks~\cite{claude-code-hooks}.
  Three hooks ship today: auto-ingesting URLs Claude fetches during a session,
  preserving conversation transcripts before context compaction, and registering
  your project directory at session start. All hooks are fail-open (a failure
  never blocks your session) and per-project configurable via
  \texttt{.claude/quarry.local.md}. You work normally; the knowledge base grows
  in the background.
\end{faqpair}

\subsection*{Internal FAQs}

\subsubsection*{Value \& Market}

\begin{faqpair}{What is the total addressable market?}\label{faq:tam}
  Claude Code reached \$1B annualized revenue by November 2025 (six months
  post-GA), then doubled weekly active users between January and February 2026,
  with annualized revenue exceeding \$2.5B~\cite{anthropic2026seriesg}. Claude
  Desktop's consumer base is larger but less precisely reported.

  The target is the power-user segment---those using Claude for complex,
  multi-document work. Conservatively 10--20\% of active Claude tool users gives
  a TAM of tens of thousands from Code alone, more with Desktop. Because Quarry
  is free, TAM is measured in adoption, not revenue. Success means becoming the
  default way Claude users make local files accessible.
\end{faqpair}

\begin{faqpair}{What evidence do we have that users want this?}\label{faq:evidence}
  \textbf{Behavioral:} Claude Desktop's Projects feature exists because users
  want to give Claude access to their documents. Users routinely ask in forums
  how to make Claude ``see'' their local files.

  \textbf{Competitive:} Rewind/Limitless, Recall, and Mem raised funding on the
  premise that personal knowledge should be searchable by AI. Rewind rebranded
  to Limitless, was acquired by Meta in December
  2025~\cite{techcrunch2025limitless}, and shut down its consumer
  product~\cite{9to5mac2025rewindshutdown}---validating demand while
  demonstrating the fragility of cloud-dependent solutions. Quarry's local-first
  architecture is resilient to this exact failure mode.

  \textbf{Adjacent:} Obsidian's plugin ecosystem shows that knowledge workers
  will invest effort to make their notes searchable by AI. Quarry eliminates the
  need for a specific note-taking app by working with any file format.
\end{faqpair}

\begin{faqpair}{Who are the competitors and why will we win?}\label{faq:competitors}
  \textbf{Claude Projects / file uploads:} The incumbent workflow. Quarry wins
  by eliminating the manual step entirely---index once, search forever, across
  all conversations.

  \textbf{Claude Code folder access:} Reads files but lacks semantic indexing.
  Quarry wins on efficiency: returns relevant chunks instead of consuming context
  window reading entire files.

  \textbf{macOS Spotlight:} System-level keyword search. Quarry adds semantic
  matching and LLM integration---Spotlight cannot find ``authentication
  logic'' in a function called {\small\texttt{verify\_credentials}}.

  \textbf{RAG frameworks (LangChain, LlamaIndex):} Require pipeline assembly,
  configuration, and Python expertise. Quarry wins on simplicity: three commands,
  zero configuration.

  \textbf{Cloud knowledge products (Mem, Notion AI):} Require accounts,
  subscriptions, and sending data to third-party servers. Rewind/Limitless was
  acquired by Meta in December 2025~\cite{techcrunch2025limitless}, its consumer
  product shut down, users stranded. Quarry wins on privacy, cost (free),
  durability (local-first, open source), and MCP-native integration.

  Quarry's MCP integration makes it a native part of how Claude works, not an
  external tool requiring explicit invocation. Competitors would need to adopt
  MCP to match this, and by then Quarry will be the established default.
\end{faqpair}

\begin{faqpair}{What is the user acquisition strategy?}\label{faq:acquisition}
  \textbf{PyPI discoverability:} Users searching for ``claude mcp'' or ``local
  RAG'' find Quarry on PyPI.

  \textbf{Claude community:} Documentation, forum posts, and word-of-mouth among
  Claude Code and Desktop users. \texttt{quarry install} auto-configures MCP,
  reducing setup friction to near zero.

  \textbf{GitHub organic:} Engineering quality (comprehensive tests, strict
  typing, clean architecture) attracts contributors and stars.

  A macOS menu bar app (\url{https://github.com/punt-labs/quarry-menubar}, in
  development) will surface Quarry to non-CLI users. The HTTP API server
  (\texttt{quarry serve}) is already shipped to support it.
\end{faqpair}

\subsubsection*{Technical}

\begin{faqpair}{What are the major technical risks?}\label{faq:tech-risks}
  \textbf{Embedding model quality (medium risk):} snowflake-arctic-embed-m-v1.5
  (768 dimensions, 512 token context) performs well for general text but may
  underperform on domain-specific content (medical, legal, financial jargon).
  Mitigation: the backend abstraction allows swapping models without code
  changes---a model upgrade is a configuration change, not an architecture change.

  \textbf{OCR accuracy on degraded scans (low risk):} RapidOCR produces
  good-enough text for semantic search but lower character accuracy than cloud
  OCR. Mitigation: Textract backend available for higher fidelity. Semantic
  search is tolerant of OCR errors because it matches by meaning, not exact
  characters.

  \textbf{LanceDB scalability (low risk):} At tens of thousands of documents,
  query latency and index size may become concerns. Mitigation: the database
  abstraction allows migration to a different backend, and named databases allow
  partitioning by project.
\end{faqpair}

\begin{faqpair}{What dependencies exist on external systems?}\label{faq:dependencies}
  \textbf{MCP protocol:} Quarry depends on the MCP specification remaining
  stable and Claude clients continuing to support MCP tool servers. Risk is low:
  Anthropic is investing heavily in MCP, and multiple third-party clients now
  support it.

  \textbf{Embedding model hosting:} The ONNX model is downloaded from Hugging
  Face Hub at install time. If removed, existing installations continue to work
  (model cached locally), but new installations would need a mirror. Mitigation:
  model is pinned to a specific git revision for reproducibility.

  \textbf{Tree-sitter grammars:} Mature, widely-used open-source projects with
  low abandonment risk.

  Quarry operates fully offline after installation.
\end{faqpair}

\begin{faqpair}{What is the development timeline and current status?}\label{faq:timeline}
  Quarry is live on PyPI. Current capabilities:

  \textbf{Shipped:} PDF ingestion (text + OCR), image OCR, spreadsheet
  ingestion (XLSX/CSV via LaTeX serialization), presentation ingestion (PPTX),
  HTML parsing, URL and sitemap crawling, text file parsing, source code
  parsing (30+ languages via Tree-sitter), semantic search with metadata
  filtering (page type, source format), directory sync, named databases,
  MCP server, CLI, HTTP API server, Claude Code plugin with ambient knowledge
  hooks (session start auto-registration, post web fetch auto-ingestion,
  pre-compact transcript preservation).

  \textbf{In progress:} macOS menu bar companion app
  (\href{https://github.com/punt-labs/quarry-menubar}{quarry-menubar}).

  \textbf{Future:} Google Drive connector, incremental re-indexing on file
  change (\texttt{quarry sync --watch}), PII detection and redaction.
\end{faqpair}

\begin{faqpair}{What is the scaling story?}\label{faq:scaling}
  Quarry is designed for personal-scale workloads: thousands to tens of thousands
  of documents on a single machine.

  Initial indexing is the main time investment. Benchmarked on an M2 MacBook
  Air, the ONNX embedding model sustains \texttildelow11 chunks/second
  (7--13 depending on batch size; see \texttt{benchmarks/embed\_throughput.py}).
  Concurrent workers provide no additional throughput due to CPU cache
  contention. A 10{,}000-file directory (\texttildelow70{,}000 chunks) takes
  roughly 1.5--2 hours. Files are queryable as they are indexed---the user does
  not wait for the full sync. Subsequent syncs are incremental (only new or
  changed files).

  Query latency is constant regardless of database size because LanceDB uses
  IVF-PQ indexing.

  Quarry does not target multi-user or server deployments. Single-user,
  single-machine scope simplifies concurrency and data privacy.
\end{faqpair}

\subsubsection*{Business}

\begin{faqpair}{What is the revenue model?}\label{faq:revenue}
  None. Quarry is free, open-source software under the MIT license. Value is
  measured in adoption and community contribution, not revenue.
\end{faqpair}

\begin{faqpair}{What are the key metrics for success?}\label{faq:metrics}
  \begin{itemize}[leftmargin=1.5em, nosep]
    \item \textbf{PyPI installs:} Monthly download count as a proxy for adoption.
          Current baseline: \texttildelow765 monthly downloads (March 2026).
          Tiered thresholds at 6 months: below 500/month triggers a go/no-go
          decision on continued development; 1{,}000/month indicates baseline
          adoption; 5{,}000/month indicates product-market fit.
    \item \textbf{GitHub stars:} Community interest signal.
          Target: 500 stars within 6 months; 2{,}000 indicates breakout.
    \item \textbf{Active MCP connections:} Users who have Quarry configured as an
          MCP server in Claude Code or Desktop (not directly measurable without
          telemetry, but inferable from support requests and community engagement).
    \item \textbf{Contributor count:} External contributors submitting PRs.
          Target: 5+ contributors within the first year.
    \item \textbf{Format coverage:} PDF, images, text, code, XLSX, PPTX, HTML
          are all shipped. Remaining targets: EPUB, email (EML/MBOX), audio
          transcription.
  \end{itemize}
\end{faqpair}

\begin{faqpair}{Why now? What has changed?}\label{faq:why-now}
  \textbf{MCP maturity:} The Model Context Protocol shipped in late 2024 and is
  now supported by Claude Code and Claude Desktop. A local tool can be a native
  part of an LLM's reasoning process, not just an external API the user invokes
  manually.

  \textbf{Local inference viability:} ONNX Runtime, quantized embedding models,
  and LanceDB have made it feasible to run a complete semantic search pipeline on
  a laptop CPU without a GPU, in under a second per query.

  \textbf{LLM power-user emergence:} A critical mass of users have moved beyond
  Q\&A to complex, multi-document tasks and are frustrated by the manual overhead
  of feeding context to their LLM.
\end{faqpair}

\begin{faqpair}{What are we not building?}\label{faq:not-building}
  \begin{itemize}[leftmargin=1.5em, nosep]
    \item \textbf{A cloud service.} Quarry will never require an account,
          subscription, or data upload. Local-first is a permanent design
          constraint, not a temporary limitation.
    \item \textbf{A multi-user platform.} No authentication, no access control,
          no shared databases. Quarry is a single-user tool for a single machine.
    \item \textbf{Media search.} Quarry does not find images by visual content
          or match audio. It extracts \emph{text} from every format and indexes
          the knowledge inside.
    \item \textbf{A note-taking app.} Quarry indexes existing files. It does not
          provide an editor, a tagging system, or a filing structure.
    \item \textbf{A general-purpose RAG framework.} No pipeline assembly, no
          pluggable retrievers, no chain-of-thought orchestration. Quarry is
          deliberately opinionated and zero-configuration.
  \end{itemize}
\end{faqpair}

\begin{faqpair}{What does failure look like?}\label{faq:failure}
  \textbf{Obscurity:} Quarry ships, works well, and nobody discovers it. After
  6 months, monthly PyPI installs plateau below 200. \textit{Response:} This is
  the most likely failure mode. Mitigation: the menu bar app (widens audience
  beyond CLI), active presence in Claude community forums, and
  \texttt{quarry install} auto-configuring MCP. If installs plateau, invest in
  content marketing before building more features.

  \textbf{First-party displacement:} Anthropic ships native file indexing in
  Claude Code or Desktop. \textit{Response:} That validates the thesis and serves
  users better than any third-party tool could. Quarry would still serve users
  who need local-only indexing, offline access, or formats Anthropic doesn't
  cover. This is a ``good problem to have'' failure.
\end{faqpair}

\begin{faqpair}{What is the next step to validate adoption?}\label{faq:next-validation}
  Run a 30-day discovery campaign targeting the Claude community (Reddit
  r/ClaudeAI, Anthropic Discord, Claude Code plugin marketplace): 500 new PyPI
  installs above the current baseline. Success validates that the discovery
  problem (see \faqref{faq:risk-value}) is solvable through community presence.
  Failure means the product needs a different entry point---likely the macOS menu
  bar app (\featureref{feat:menu-bar})---before community marketing can generate
  traction.
\end{faqpair}

\section*{Risk Assessment}

\begin{faqpair}{Value --- Will users adopt this?}\label{faq:risk-value}
  \textcolor{RiskAmber}{\textbf{Medium risk.}} Demand is proven: Claude Projects
  exists because users asked for a way to use their local files with AI. The risk
  is not demand---it is \emph{discovery and activation}. Nobody searches for
  ``MCP-based local semantic search.'' They search for ``how to use my files with
  Claude without uploading them.'' Quarry must be present where those questions
  are asked---Reddit, Claude community forums, Stack Overflow---with a
  one-sentence answer: ``Stop uploading files. Install Quarry, index once, search
  forever.''

  Activation risk is real. A user who indexes one file will not see the value.
  The ``wow'' moment---Claude retrieving a document the user forgot they
  had---requires enough indexed content to make semantic search surprising.
  Onboarding must encourage bulk ingestion (directory registration) over
  single-file ingest.

  The macOS menu bar app is the highest-leverage roadmap item: it widens the
  audience beyond CLI users and makes retrieval visible, creating shareable
  moments that drive word-of-mouth.
\end{faqpair}

\begin{faqpair}{Usability --- Can users figure out how to use it?}\label{faq:risk-usability}
  \textcolor{RiskAmber}{\textbf{Medium risk.}} For CLI-native users, three
  commands to value: \texttt{uv tool install}, \texttt{quarry install},
  \texttt{quarry ingest}. Auto-configures MCP. No accounts, no API keys, no
  configuration files. Once set up, Claude invokes Quarry automatically.

  The product currently requires terminal comfort. For the non-CLI audience
  identified as a growth target, usability risk is real. The macOS menu bar app
  (in development) is the primary mitigation: it reduces setup to a native macOS
  install with no terminal required. Until then, Quarry is accessible only to
  developers and technical power users.
\end{faqpair}

\begin{faqpair}{Feasibility --- Can we build it?}\label{faq:risk-feasibility}
  \textcolor{RiskGreen}{\textbf{Low risk.}} The product is live on PyPI. Core
  pipeline (ingest, embed, search, MCP server) is shipped across 10 document
  formats with 794 passing tests and strict type checking. Remaining roadmap
  items (menu bar app, Google Drive connector, \texttt{sync --watch}) are
  additive features, not architectural risks.
\end{faqpair}

\begin{faqpair}{Viability --- Does it work as a project?}\label{faq:risk-viability}
  \textcolor{RiskGreen}{\textbf{Low risk.}} No revenue model to validate.
  Infrastructure cost is zero. Development cost is the maintainer's time. MIT
  license eliminates legal risk. The project is compatible with Anthropic's
  ecosystem strategy.
\end{faqpair}

\section*{Feature Appendix}

Scope boundary. Every feature is explicitly categorized to prevent scope creep
and force prioritization decisions.

\subsection*{Core (Shipped)}

Shipped features that form the core product.

\begin{enumerate}[nosep,leftmargin=2.5em]
  \featureitem{Local semantic search}{Core product: index local files and retrieve passages by meaning across 10+ document formats.}\label{feat:semantic-search}
  \featureitem{MCP server}{Exposes search to Claude Code and Claude Desktop as a native tool, enabling automatic retrieval during conversations.}\label{feat:mcp-server}
  \featureitem{Directory registration and sync}{Bulk ingestion workflow: register a directory once, sync incrementally as files change.}\label{feat:directory-sync}
  \featureitem{Local OCR}{Extract text from scanned documents and images on-device with no cloud account required.}\label{feat:local-ocr}
  \featureitem{Named databases}{Organize separate knowledge bases by project or domain to keep search results relevant.}\label{feat:named-databases}
  \featureitem{One-command install}{A single \texttt{quarry install} auto-configures MCP for Claude Code and Claude Desktop with zero manual setup.}\label{feat:one-command-install}
  \featureitem{Claude Code plugin}{Plugin with three ambient knowledge hooks: auto-register project directory on session start, auto-ingest URLs Claude fetches (post web fetch), and preserve conversation transcripts before context compaction (pre-compact). All hooks fail-open, non-blocking, per-project configurable.}\label{feat:claude-plugin}
\end{enumerate}

\subsection*{Must Do Next}

Highest-leverage item for addressing adoption risks identified in
\faqref{faq:risk-value} and \faqref{faq:risk-usability}.

\begin{enumerate}[nosep,leftmargin=2.5em]
  \featureitem{macOS menu bar app}{Surfaces Quarry to non-CLI users and makes retrieval visible, expanding adoption beyond developers. Identified as the highest-leverage roadmap item in \faqref{faq:risk-value}.}\label{feat:menu-bar}

Roadmap features that expand the product but are not required for core value.

\begin{enumerate}[nosep,leftmargin=2.5em]

  \featureitem{General-purpose RAG framework}{No pipeline assembly, no pluggable retrievers, no chain-of-thought orchestration---deliberately opinionated and zero-configuration.}\label{feat:no-rag-framework}
