\section*{Press Release}

PORTLAND, OR --- September 2026 --- Today, Punt Labs announced Quarry, a free,
open-source tool that gives Claude persistent memory and instant access to
everything on your computer. Quarry indexes PDFs, scanned documents,
spreadsheets, presentations, web pages, notes, source code, and images into a
local semantic search engine, then exposes that knowledge to Claude Code and
Claude Desktop through the Model Context
Protocol~\cite{aitoolanalysis2026claudeplugins}. As a Claude Code plugin,
Quarry goes further: every web page Claude researches, every debugging
conclusion reached, every conversation transcript is automatically captured
into the knowledge base before it can be lost to context compaction. Unlike
hosted context platforms that require cloud accounts and team infrastructure,
Quarry runs entirely on the developer's machine, is open source under the MIT
license, and works with the Claude Code plugin system that has grown to over
9{,}000 extensions (see \faqref{faq:tam}).

\prsection{Problem}

Developers who use Claude Code for serious, multi-session work---refactoring
systems, debugging production issues, designing architectures---lose everything
they learn at the end of every conversation. A developer spends forty-five
minutes with Claude tracking down a connection timeout, reads three Stack
Overflow answers, identifies the root cause, and ships the fix. Two weeks later,
the same class of issue appears in a different service. Claude starts from
zero---searches the web again, reads the same answers, discovers the same root
cause. The developer recognizes the pattern but Claude does not, because Claude
has no memory of the previous session.

The problem extends beyond debugging. People who use Claude for research,
analysis, or writing hit the same wall. They spend 5--15 minutes gathering
context before they can ask their real question. They upload PDFs to Projects.
They copy-paste from old documents. They point Claude Code at directories and
hope the right files get picked up. When the context window fills, they start
over. A developer with ten years of reference PDFs and design documents cannot
make any of that knowledge available to Claude without manually curating it
first.

The knowledge is generated. The work is done. And then it vanishes
(see \faqref{faq:evidence}).

\prsection{Solution}

Quarry works at two levels: as a search engine for your files, and as a plugin
that captures knowledge automatically.

\textbf{Search engine.} Install with \texttt{uv tool install punt-quarry} and
\texttt{quarry install}. Point it at your
files---\texttt{quarry ingest report.pdf}, or register an entire directory with
\texttt{quarry register \texttildelow/Documents/research}. From that moment,
Claude can search your indexed knowledge by meaning rather than keywords. Every
format is handled automatically: text PDFs are extracted, scanned documents and
images are OCR'd locally, spreadsheets are serialized to preserve tabular
structure, presentations are split slide-by-slide, HTML pages are stripped of
boilerplate, and source code is parsed into semantic units. The result: chunks
Claude can retrieve, with metadata indicating the document, page, content type,
and source format.

\textbf{Ambient knowledge plugin.} As a Claude Code plugin, Quarry captures
knowledge passively through hooks. When Claude fetches a web page, a hook
queues the URL for background ingestion. Before context compaction, a hook
captures the conversation transcript. Your project directory is registered the
moment a session starts. Every hook is fail-open and non-blocking: you work
normally, and the knowledge base grows from the natural flow of every session.
Per-project configuration via \texttt{.punt-labs/quarry/config.md} lets you
selectively disable individual hooks.

Because Quarry runs entirely on your machine, your documents never leave your
laptop. No cloud service, no API key, no subscription.

\prsection{Getting Started}

Getting started requires no accounts, no API keys, and takes under five
minutes:

\begin{enumerate}[leftmargin=2em]
  \item \textbf{Install:} {\small\texttt{uv tool install punt-quarry \&\& quarry install}}
        downloads the embedding model, installs the daemon, and configures
        Claude Code and Claude Desktop automatically.
  \item \textbf{Ingest:} \texttt{quarry register \texttildelow/Documents/work
        \&\& quarry sync} indexes an entire directory. Or ingest individual files
        with \texttt{quarry ingest report.pdf}.
  \item \textbf{Work normally.} Open Claude Code or Claude Desktop and ask a
        question. Claude calls Quarry's search tool automatically. The plugin
        hooks capture web research and session transcripts in the background.
\end{enumerate}

\subsection*{External FAQs}

\begin{faqpair}{What is Quarry and who is it for?}\label{faq:what}
  Quarry is a local semantic search engine and Claude Code plugin that gives
  your coding sessions persistent memory. It indexes files on your hard drive
  across 10+ formats, captures web research and conversation transcripts
  automatically via plugin hooks, and makes all of it searchable by meaning.
  It is for developers who use Claude Code or Claude Desktop for multi-session
  work and are tired of Claude re-researching topics it has already
  investigated, or manually uploading files before every session.
\end{faqpair}

\begin{faqpair}{How is this different from uploading files to Claude Projects?}\label{faq:vs-projects}
  Projects require you to select and upload files before every conversation, the
  context window limits how many you can include, and when you start a new
  conversation, you start from scratch. Quarry inverts this: index your files
  once, and Claude searches them on demand during any conversation. You don't
  decide what's relevant---Claude does. There is no context window limit because
  Quarry returns only the relevant passages, not entire files.
\end{faqpair}

\begin{faqpair}{How is this different from pointing Claude Code at a folder?}\label{faq:vs-folders}
  When Claude Code reads a directory, it reads files sequentially, consuming
  context window with every file it opens. For a folder with 50 documents, Claude
  Code may exhaust its context before finding the relevant passage. Quarry
  pre-indexes the content and returns only matching chunks, leaving the context
  window for reasoning rather than reading.
\end{faqpair}

\begin{faqpair}{How is this different from CLAUDE.md or memory files?}\label{faq:vs-claude-md}
  CLAUDE.md is static instructions that you write and maintain manually. It
  tells Claude \emph{how} to behave. Quarry captures \emph{what Claude
  learned}---web pages, debugging conclusions, design rationale---and makes it
  searchable by meaning. You do not write or maintain the knowledge base; it
  builds itself from every session. CLAUDE.md says ``use ruff for linting.''
  Quarry remembers that the last time you debugged a ruff configuration issue,
  the root cause was a conflicting pyproject.toml section, and the Stack
  Overflow answer that explained it.
\end{faqpair}

\begin{faqpair}{What file formats does Quarry support?}\label{faq:formats}
  PDF (both text and scanned/image pages), images (PNG, JPG, TIFF, BMP, WebP),
  spreadsheets (XLSX, CSV), presentations (PPTX), HTML, text files (Markdown,
  LaTeX, plain text, DOCX), and source code in 30+ languages. Every format is
  converted to text optimized for LLM consumption. Spreadsheets and presentation
  tables are serialized to LaTeX to preserve tabular structure. Scanned documents
  and images are OCR'd locally---no cloud account needed.
\end{faqpair}

\begin{faqpair}{How does the ambient capture work?}\label{faq:how-it-works}
  Quarry uses Claude Code's hook system~\cite{claude-code-hooks}---the same
  event-driven architecture used by plugins like Entire.io for code provenance.
  Three hooks ship today:

  \begin{itemize}[leftmargin=1.5em, nosep]
    \item \textbf{Session start} --- auto-registers the project directory so
          new files are indexed on sync, and injects a knowledge base summary
          into Claude's context.
    \item \textbf{Post web fetch} --- every URL Claude fetches is auto-ingested
          into the knowledge base for later semantic search.
    \item \textbf{Pre-compact} --- conversation transcript is captured before
          context compaction, preserving decisions and discoveries that would
          otherwise be lost.
  \end{itemize}

  All hooks write to a local staging directory and return
  immediately---they never block the conversation. A background process handles
  embedding and indexing. The hooks are fail-open: if Quarry is unavailable,
  Claude Code works exactly as before. Per-project configuration via
  \texttt{.punt-labs/quarry/config.md} lets you selectively disable individual
  hooks.
\end{faqpair}

\begin{faqpair}{Does my data leave my computer?}\label{faq:privacy}
  No. Quarry runs entirely on your machine. The vector database (LanceDB), the
  embedding model (snowflake-arctic-embed-m-v1.5 via ONNX Runtime), the OCR
  engine (RapidOCR), and all hooks execute locally. No data is sent to any
  cloud service. Captured web pages are re-fetched and stored locally, not
  forwarded through a proxy.
\end{faqpair}

\begin{faqpair}{How much does it cost?}\label{faq:cost}
  Free. Quarry is open source under the MIT license. There is no paid tier, no
  usage limits, and no telemetry. The embedding model download is approximately
  500\,MB; after that, everything runs offline.
\end{faqpair}

\begin{faqpair}{Can I use Quarry with tools other than Claude?}\label{faq:other-tools}
  Quarry exposes a standard MCP server over stdio and an HTTP API with
  WebSocket support. Any MCP-compatible client can use it. Today that includes
  Claude Code and Claude Desktop. As other LLM tools adopt MCP, Quarry will
  work with them without modification. The CLI also works standalone for
  direct search without any LLM.
\end{faqpair}

\begin{faqpair}{How does search quality compare to keyword search (Spotlight, grep)?}\label{faq:search-quality}
  Quarry finds passages by meaning, not by matching words. Searching for
  ``authentication logic'' finds a function called \texttt{verify\_credentials}
  even though the words don't overlap. This is the same retrieval approach used
  by commercial RAG systems, running locally with a 768-dimension embedding
  model.
\end{faqpair}

\begin{faqpair}{Do I have to manually index everything?}\label{faq:passive-capture}
  No. The CLI and MCP server support explicit commands, but the Claude Code
  plugin captures knowledge passively through hooks~\cite{claude-code-hooks}.
  Three hooks ship today: auto-ingesting URLs Claude fetches during a session,
  preserving conversation transcripts before context compaction, and registering
  your project directory at session start. All hooks are fail-open
  (a failure never blocks your session) and per-project configurable via
  \texttt{.punt-labs/quarry/config.md}. You work normally; the knowledge base grows
  in the background.
\end{faqpair}

\subsubsection*{Value \& Market}

\begin{faqpair}{What is the total addressable market?}\label{faq:tam}
  Claude Code had over 115{,}000 developers by mid-2025~\cite{ppcland2025claudecode115k}
  and has been growing rapidly, with Anthropic reporting doubled weekly active
  users and annualized revenue exceeding \$2.5B by early
  2026~\cite{anthropic2026seriesg}. The plugin ecosystem grew from zero to
  9{,}000+ extensions in under five months~\cite{startuphub2025claudeplugins}.
  The broader AI-assisted coding market includes users of Cursor, GitHub
  Copilot, and Windsurf, totaling millions of developers.

  The target segment is \emph{multi-session power users}---developers who use
  AI coding tools for complex work spanning days or weeks, not single-turn
  Q\&A. The Qodo survey found that 41--52\% of developers report
  context-related frustration~\cite{qodo2025aicodequality}. If even a quarter
  of frustrated users are multi-session Claude Code users who would benefit
  from passive capture, the segment is tens of thousands. Because Quarry is
  free, TAM is measured in adoption, not revenue. Success means becoming the
  default way Claude Code sessions persist knowledge.
\end{faqpair}

\begin{faqpair}{What evidence do we have that developers want this?}\label{faq:evidence}
  Four categories of evidence:

  \textbf{Survey data:} The Qodo ``State of AI Code Quality'' survey found
  that context-related frustration affects 41--52\% of developers (increasing
  with seniority), and ``improved contextual understanding'' is the single
  most-requested AI improvement at 26\% of votes. The Stack Overflow 2025
  survey (49{,}000+ respondents) found 81\% have security/privacy concerns
  about AI data use~\cite{stackoverflow2025survey}---local-first architecture
  directly addresses this.

  \textbf{Market signal:} Multiple startups are building products for this
  exact problem. SageOx~\cite{sageox2026} positions itself as ``context
  infrastructure for human-agent collaboration.'' Entire.io, founded by former
  GitHub CEO Thomas Dohmke with \$60M in seed
  funding~\cite{techcrunch2026entirefunding}, captures the reasoning behind
  AI-generated code via a git shadow branch~\cite{entireio2026github}. Both
  validate that knowledge loss between agent sessions is a problem worth
  solving.

  \textbf{Behavioral:} Claude Code's own memory features (CLAUDE.md,
  auto-memory) exist because developers asked for session persistence. These
  features are static and manually maintained; they capture instructions, not
  learned knowledge. The gap between ``remember my preferences'' and ``remember
  what we researched'' is the gap Quarry fills.

  \textbf{Adjacent:} The cautionary example is Rewind/Limitless---a personal
  knowledge capture tool that raised \$33M+, was acquired by Meta in December
  2025, and shut down both its Pendant hardware and Rewind desktop
  app~\cite{techcrunch2025limitless,9to5mac2025rewindshutdown}. The demand was
  real; the cloud-dependent model was fragile. Quarry's local-first
  architecture is resilient to this failure mode.
\end{faqpair}

\begin{faqpair}{Who are the competitors and why will we win?}\label{faq:competitors}
  \textbf{SageOx:} Hosted context platform for enterprise
  teams~\cite{sageox2026}. Captures agent sessions and team discussions as
  structured artifacts. Quarry is not competing for enterprise teams---it
  serves individuals and small teams with a local-first, open-source tool that
  requires no accounts or infrastructure.

  \textbf{Entire.io:} Local-first session recording for individual
  developers~\cite{entireio2026github}. Uses a git shadow branch to store
  session metadata alongside code. Quarry's differentiator is semantic
  search---not just recording what happened, but making it searchable by
  meaning across sessions, formats, and topics.

  \textbf{Windsurf Cascade Memories:} Built-in cross-conversation persistent
  context in the Windsurf AI code editor~\cite{codeant2025windsurf}. However,
  it is cloud-hosted, Windsurf-specific, and does not support local-first data
  or cross-tool search. Quarry works with any Claude Code session and stores
  everything locally.

  \textbf{CLAUDE.md / auto-memory:} Built-in Claude Code features for session
  persistence. These capture static instructions and preferences, not dynamic
  knowledge. They are complementary, not competitive---CLAUDE.md says how to
  work; Quarry remembers what was learned.

  \textbf{Claude Projects / file uploads:} The incumbent workflow. Quarry wins
  by eliminating the manual step entirely---index once, search forever, across
  all conversations.

  \textbf{Platform risk:} The most dangerous competitive response is Anthropic
  building ambient memory natively into Claude Code. Quarry's residual value in
  that scenario: (1)~cross-tool portability---the knowledge base works with any
  MCP client, the CLI, and the HTTP API; (2)~user-owned data in an open format
  (LanceDB, not a proprietary store); (3)~30+ format indexing beyond what a
  native feature would likely support; (4)~open source, auditable, and
  extensible.
\end{faqpair}

\begin{faqpair}{What is the user acquisition strategy?}\label{faq:acquisition}
  Quarry already has an installed base from the search tool (PyPI downloads,
  GitHub presence). The ambient knowledge features are delivered as a plugin
  update---existing users get the new capabilities automatically. New users
  discover Quarry through PyPI, GitHub, the Claude Code plugin marketplace,
  and Claude community forums.

  \textbf{PyPI discoverability:} Users searching for ``claude mcp'' or ``local

  \textbf{Claude community:} Documentation, forum posts, and word-of-mouth among

  \textbf{GitHub organic:} Engineering quality (comprehensive tests, strict
  typing, clean architecture) attracts contributors and stars.
\end{faqpair}

\subsubsection*{Technical}

\begin{faqpair}{What are the major technical risks?}\label{faq:tech-risks}
  \textbf{Embedding model quality (medium risk):} snowflake-arctic-embed-m-v1.5
  (768 dimensions, 512 token context) performs well for general text but may
  underperform on domain-specific content (medical, legal, financial jargon).
  Mitigation: the backend abstraction allows swapping models without code
  changes---a model upgrade is a configuration change, not an architecture change.

  \textbf{OCR accuracy on degraded scans (low risk):} RapidOCR produces
  good-enough text for semantic search but lower character accuracy than
  commercial OCR. Mitigation: semantic search is tolerant of OCR errors
  because it matches by meaning, not exact characters.

  \textbf{LanceDB scalability (low risk):} At tens of thousands of documents,
  query latency and index size may become concerns. Mitigation: the database
  abstraction allows migration to a different backend, and named databases allow
  partitioning by project.

  \textbf{Hook latency (low risk):} All hooks must return fast to avoid
  blocking Claude Code. The hooks write to files and return
  immediately---no embedding or network calls in the hook path. Cold start
  for the hook dispatcher is 0.24s (stdlib-only imports, no heavy
  dependencies).
\end{faqpair}

\begin{faqpair}{What dependencies exist on external systems?}\label{faq:dependencies}
  \textbf{MCP protocol:} Quarry depends on the MCP specification remaining
  stable and Claude clients continuing to support MCP tool servers. Risk is low:
  Anthropic is investing heavily in MCP, and multiple third-party clients now
  support it.

  \textbf{Claude Code hook system:} The ambient capture layer depends on Claude
  Code's hook events (PostToolUse, PreCompact, SessionStart) remaining stable
  and available to plugins. Risk is low: the hook system is actively developed
  and used by multiple shipping plugins~\cite{claude-code-hooks}.

  \textbf{Embedding model hosting:} The ONNX model is downloaded from Hugging
  Face Hub at install time. If removed, existing installations continue to work
  (model cached locally), but new installations would need a mirror. Mitigation:
  model is pinned to a specific git revision for reproducibility.

  \textbf{Tree-sitter grammars:} Mature, widely-used open-source projects with
  low abandonment risk.

  Quarry operates fully offline after installation.
\end{faqpair}

\begin{faqpair}{What is the development timeline and current status?}\label{faq:timeline}
  Quarry is live on PyPI. Current capabilities:

  \textbf{Shipped:} PDF ingestion (text + local OCR), image OCR, spreadsheet
  ingestion (XLSX/CSV), presentation ingestion (PPTX), HTML parsing, URL and
  sitemap crawling, text file parsing, source code parsing (30+ languages via
  Tree-sitter), semantic search with metadata filtering, directory sync with
  \texttt{.gitignore}/\texttt{.quarryignore} support, named databases, MCP
  server with 11 tools, CLI, HTTP API with WebSocket MCP endpoint, Claude
  Desktop extension (.mcpb), Claude Code plugin with three hooks (session
  start auto-registration, post web fetch auto-ingestion, pre-compact
  transcript preservation), daemon architecture (launchd/systemd) with
  mcp-proxy for shared-process MCP, researcher subagent for combined local
  + web search, CLAUDE.md injection for agent discovery. Extensive automated
  tests, strict type checking (mypy + pyright).

  \textbf{Roadmap:} Knowledge recall system (session briefing, local-first
  search nudge), \texttt{quarry sync --watch} for live filesystem monitoring,
  PII detection and redaction, Google Drive connector.
\end{faqpair}

\begin{faqpair}{What is the scaling story?}\label{faq:scaling}
  Quarry is designed for personal-scale workloads: thousands to tens of thousands
  of documents on a single machine.

  Initial indexing is the main time investment. Benchmarked on an M2 MacBook
  Air, the ONNX embedding model sustains \texttildelow11 chunks/second.
  A 10{,}000-file directory (\texttildelow70{,}000 chunks) takes roughly
  1.5--2 hours. Files are queryable as they are indexed---the user does not
  wait for the full sync. Subsequent syncs are incremental (only new or
  changed files).

  Query latency is constant regardless of database size because LanceDB uses
  IVF-PQ indexing.

  Quarry does not target multi-user or server deployments. Single-user,
  single-machine scope simplifies concurrency and data privacy. The one
  exception is the Fly.io deployment that serves the punt-labs.com chat
  widget's RAG backend.
\end{faqpair}

\begin{faqpair}{What happens as the knowledge base grows?}
  LanceDB is designed for datasets far larger than a personal knowledge base.
  Search latency remains sub-second up to millions of vectors; a knowledge base
  of 10{,}000--50{,}000 documents (years of heavy use) is well within
  comfortable range. Disk usage scales at approximately 1--5\,MB per 100
  documents for embeddings and metadata. Raw captured content (web pages,
  transcripts) adds more but is bounded by dedup logic---the same URL is not
  re-ingested.

  If disk usage becomes a concern, the existing \texttt{quarry delete} command
  lets users prune old or low-value collections.
\end{faqpair}

\subsubsection*{Business}

\begin{faqpair}{What is the revenue model?}\label{faq:revenue}
  None. Quarry is free, open-source software under the MIT license. Value is
  measured in adoption and community contribution, not revenue.
\end{faqpair}

\begin{faqpair}{What are the key metrics for success?}\label{faq:metrics}
  \begin{itemize}[leftmargin=1.5em, nosep]
    \item \textbf{PyPI installs:} Monthly download count as a proxy for adoption.
          Current baseline: \texttildelow765 monthly downloads (March 2026).
          Tiered thresholds at 6 months: below 500/month triggers a go/no-go
          decision on continued development; 1{,}000/month indicates baseline
          adoption; 5{,}000/month indicates product-market fit.
    \item \textbf{GitHub stars:} Community interest signal.
          Target: 500 stars within 6 months; 2{,}000 indicates breakout.
    \item \textbf{Active MCP connections:} Users who have Quarry configured as an
          MCP server in Claude Code or Desktop (not directly measurable without
          telemetry, but inferable from support requests and community engagement).
    \item \textbf{Contributor count:} External contributors submitting PRs.
          Target: 5+ contributors within the first year.
    \item \textbf{Format coverage:} PDF, images, text, code, XLSX, PPTX, HTML
          are all shipped. Remaining targets: EPUB, email (EML/MBOX), audio
          transcription.
  \end{itemize}
\end{faqpair}

\begin{faqpair}{Why now? What has changed?}\label{faq:why-now}
  Three shifts converged:

  \textbf{MCP maturity:} The Model Context Protocol shipped in late 2024 and is
  now supported by Claude Code and Claude Desktop. A local tool can be a native
  part of an LLM's reasoning process, not just an external API the user invokes
  manually.

  \textbf{Claude Code plugin ecosystem maturation.} The hook system
  (PostToolUse, PreToolUse, PreCompact, SessionStart) is now stable and used
  by shipping plugins~\cite{claude-code-hooks}. The ecosystem grew to 9{,}000+
  plugins in under five months~\cite{startuphub2025claudeplugins}. The
  infrastructure for ambient behavior exists; Quarry applies it to knowledge
  persistence.

  \textbf{Local inference viability:} ONNX Runtime, quantized embedding models,
  and LanceDB have made it feasible to run a complete semantic search pipeline on
  a laptop CPU without a GPU, in under a second per query.
\end{faqpair}

\begin{faqpair}{What are we not building?}\label{faq:not-building}
  \begin{itemize}[leftmargin=1.5em, nosep]
    \item \textbf{A cloud service.} Quarry will never require an account,
          subscription, or data upload. Local-first is a permanent design
          constraint, not a temporary limitation.
    \item \textbf{A multi-user platform.} No authentication, no access control,
          no shared databases. Quarry is a single-user tool for a single machine.
    \item \textbf{Media search.} Quarry does not find images by visual content
          or match audio. It extracts \emph{text} from every format and indexes
          the knowledge inside.
    \item \textbf{A note-taking app.} Quarry indexes existing files. It does not
          provide an editor, a tagging system, or a filing structure.
    \item \textbf{A general-purpose RAG framework.} No pipeline assembly, no
          pluggable retrievers, no chain-of-thought orchestration. Quarry is
          deliberately opinionated and zero-configuration.
    \item \textbf{Structured knowledge extraction.} Quarry indexes raw content
          and lets semantic search surface what is relevant. It does not extract
          summaries, decisions, or intent from captured content.
    \item \textbf{Code provenance tracking.} Quarry does not add metadata to
          git commits or link commits to sessions.
  \end{itemize}
\end{faqpair}

\begin{faqpair}{What does failure look like?}\label{faq:failure}
  Three plausible failure scenarios:

  \textbf{Obscurity:} Quarry ships, works well, and nobody discovers it. After
  6 months, monthly PyPI installs plateau below 200. \textit{Response:} This is
  the most likely failure mode. Mitigation: active presence in Claude community
  forums, and \texttt{quarry install} auto-configuring MCP. If installs plateau,
  invest in content marketing before building more features.

  \textbf{First-party displacement:} Anthropic ships native file indexing or
  session memory in Claude Code. \textit{Response:} That validates the thesis
  and serves users better than any third-party tool could. Quarry would still
  serve users who need local-only indexing, offline access, or formats Anthropic
  doesn't cover.

  \textbf{Capture noise exceeds signal.} The knowledge base fills with
  low-value web pages and irrelevant transcript fragments, degrading search
  quality. \textit{Response:} Start conservative (web research + compaction
  only). If signal-to-noise is poor, add quality filtering---URL pattern
  exclusions, minimum page length, dedup by content similarity.
\end{faqpair}

\begin{faqpair}{What is the next step to validate adoption?}\label{faq:next-validation}
  Build the knowledge recall system: a session-start briefing that tells Claude
  what is in the knowledge base, and a pre-web-search nudge that suggests
  checking local knowledge first. These hooks are read-only, fail-open, and
  can be disabled in one command. If the recall mechanism demonstrably reduces
  redundant web searches in real usage, the ambient capture hooks become more
  valuable. If not, the capture is still useful for manual search via
  \texttt{/find}.
\end{faqpair}

\vspace{1em}

\textbf{Value risk} is the dominant concern. The thesis is that local semantic
search combined with passive knowledge capture changes how developers work
with AI coding tools---that your LLM should have access to everything you
know, and that the knowledge generated during sessions should compound instead
of evaporating. Discovery is the immediate challenge: users must find Quarry
and invest enough initial indexing to experience the ``wow'' moment of Claude
finding a document they forgot they had. The planned recall system---session
briefings and pre-web-search nudges---will test whether automatic surfacing of
local knowledge changes behavior, but the capture hooks already deliver value
for manual search.

\section*{Feature Appendix}

Defines the scope boundary for Quarry. Features are categorized to prevent
scope creep and force prioritization.

\subsection*{Core (Shipped)}

Shipped features that form the core product.

\begin{enumerate}[nosep,leftmargin=2.5em]
  \featureitem{Local semantic search}{Core product: index local files and retrieve passages by meaning across 10+ document formats.}\label{feat:semantic-search}
  \featureitem{MCP server}{Exposes search to Claude Code and Claude Desktop as a native tool, enabling automatic retrieval during conversations. 11 tools covering search, ingestion, management, and database operations.}\label{feat:mcp-server}
  \featureitem{Directory registration and sync}{Bulk ingestion workflow: register a directory once, sync incrementally as files change. Respects \texttt{.gitignore} and \texttt{.quarryignore}.}\label{feat:directory-sync}
  \featureitem{Local OCR}{Extract text from scanned documents and images on-device via RapidOCR with no cloud account required.}\label{feat:local-ocr}
  \featureitem{Named databases}{Organize separate knowledge bases by project or domain to keep search results relevant.}\label{feat:named-databases}
  \featureitem{One-command install}{A single \texttt{quarry install} auto-configures MCP for Claude Code and Claude Desktop, downloads mcp-proxy, and installs the daemon.}\label{feat:one-command-install}
  \featureitem{Claude Code plugin}{Plugin with three hooks: auto-register project directory on session start, auto-ingest URLs Claude fetches (post web fetch), and preserve conversation transcripts before context compaction (pre-compact). All hooks fail-open, non-blocking, per-project configurable.}\label{feat:claude-plugin}
  \featureitem{Daemon architecture}{Single background process (\texttt{quarry serve}) loads the embedding model once and serves all sessions via WebSocket MCP endpoint. Lightweight mcp-proxy binary bridges MCP stdio to the daemon, eliminating per-session Python startup.}\label{feat:daemon}
  \featureitem{Researcher agent}{Plugin subagent that combines local Quarry search with web research. Searches locally first, web for gaps, and auto-ingests valuable findings so research compounds across sessions.}\label{feat:researcher}
\end{enumerate}

\subsection*{Should Do}

Roadmap features that expand the product but are not required for core value.

\begin{enumerate}[nosep,leftmargin=2.5em]
  \featureitem{Knowledge recall system}{Session-start briefing and pre-web-search nudge that surface relevant local knowledge before Claude searches the web. Closes the learn/recall loop.}\label{feat:knowledge-recall}
  \featureitem{Live filesystem monitoring}{Automatic re-indexing via \texttt{sync --watch} eliminates the manual sync step after file changes.}\label{feat:sync-watch}
  \featureitem{Google Drive connector}{Extends ingestion beyond local files to cloud-hosted documents without requiring manual download.}\label{feat:google-drive}
  \featureitem{PII detection and redaction}{Prevents sensitive personal information from entering the search index, addressing privacy-conscious users.}\label{feat:pii-redaction}
\end{enumerate}

\subsection*{Won't Do}

Explicitly excluded. Naming what Quarry will not build prevents scope creep.

\begin{enumerate}[nosep,leftmargin=2.5em]
  \featureitem{Multi-user platform}{No authentication, no access control, no shared databases---Quarry is a single-user tool for a single machine.}\label{feat:no-multi-user}
  \featureitem{Media search by visual or audio content}{Quarry extracts text from every format; it does not match images by visual similarity or search audio by sound.}\label{feat:no-media-search}
  \featureitem{Note-taking or filing system}{Quarry indexes existing files---it does not provide an editor, tagging system, or organizational structure.}\label{feat:no-note-taking}
  \featureitem{General-purpose RAG framework}{No pipeline assembly, no pluggable retrievers, no chain-of-thought orchestration---deliberately opinionated and zero-configuration.}\label{feat:no-rag-framework}
  \featureitem{Hosted web UI}{Team browsing of shared knowledge is outside scope.}\label{feat:no-web-ui}
  \featureitem{Structured knowledge extraction}{Extracting summaries, decisions, and intent from raw content adds complexity and is lossy---semantic search over raw content is sufficient.}\label{feat:no-extraction}
  \featureitem{Git commit metadata}{Adding checkpoint trailers or session IDs to git commits is outside scope.}\label{feat:no-git-meta}
\end{enumerate}
